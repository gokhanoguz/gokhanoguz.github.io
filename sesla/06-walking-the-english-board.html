Walking the English Board, One Screenshot at a Time

The Phase 9c audit shipped a few days ago: 709 tests green, English locale at parity with Turkish on paper. But "passes the test suite" and "looks right on a real device" are not the same thing. Today was a slow walk through every English category in the iOS simulator, screenshot by screenshot, fixing every defect that showed up.

Lowercase Is the AAC Convention

First thing that jumped out: every English label was Title Case. Boy, Girl, Apple. Every commercial AAC app I checked uses lowercase. There's a reason — lowercase letterforms have more distinctive shapes, which helps early readers and AAC users with visual processing differences. Capitals all look like rectangles.

A bulk lowercase script is trivial. A bulk lowercase script that doesn't break I, Sunday, Easter, the alphabet (A, B, C…), or proper nouns is slightly less trivial. The first version stripped everything and I had to revert and add a skip list. AAC apps care about edge cases more than most software does.

The Default Board Was Wrong

Switched the app to English. Opened Boards. The default board that loaded was Turkish. Several tiles on it were marked hide:en, so they showed up as gray holes. Confusing for any English user.

The board provider was returning "the first default board" regardless of language. Made it language-aware: if you've picked English, give me the board whose ID ends in en-basic-communication; if Turkish, give me tr-temel-iletisim. Three lines, instant fix, but you only catch it by actually flipping the language toggle and looking.

Baked Text Was Hiding Everywhere

AAC symbols traditionally have the word baked into the SVG, on top of the picture. Sesla's tile already shows the word as a label below the icon, so the baked word is redundant — every tile reads "he he" or "need need". Fine on Turkish, where I'd been disciplined about it. Less fine on English.

Two flavors of bug:

  • Four icons had Turkish baked text (e.g. "DUR" on the stop sign) showing through on the English board. Stripped.
  • 22 icons had English baked text duplicating their label. Stripped.

The trick was not stripping everything. Some icons are text-only by design — the symbol for and is just the word "and" in a colored rectangle. Strip that and you're left with an empty colored rectangle. The fix script checks each SVG for non-text glyph content (paths, circles, polygons) before deciding it's safe to strip. Text-only cards survive; redundant overlays die.

Eleven Icons Were Pointing at Nothing

Some tiles rendered as broken-image gray placeholders: boy, neighbor, we, they, good night, bored, calm, to throw, and a few more. The SVG files existed on disk. The JSON just had empty image_path fields pointing at nothing. Drift had happily seeded a "this symbol has no picture" record for each one.

Eleven empty paths populated. Seed version bumped so existing installs re-seed on next launch. This kind of bug is invisible to the test suite — every test passed because the schema was valid; the data was just incomplete.

The Seed Version Trick

A note on how data fixes ship to users who already have the app installed. The bundled symbol JSON has a version number. The app stores the last-seeded version in user prefs. On launch, if the bundled version is higher, it re-seeds the symbol table (preserving user-created custom symbols). Bumped from v34 → v37 across today's three commits.

Boards work differently — they only seed once, ever, to avoid clobbering user-edited boards. So board changes need a different mechanism for in-place updates. Today's fixes were all symbol-level so the version bump handles it.

The Screenshot Loop

My workflow today was: ask Copilot to take a screenshot, look at it, identify everything wrong, log it, scroll to the next category, repeat. Twelve categories, twelve passes. Then a single batch fix at the end covering everything that wasn't a one-off.

Tests catch logic bugs. Eyes catch design bugs. Both matter equally for an app whose entire purpose is visual communication.

What's Next

Tomorrow: English morphology gets the same depth treatment Turkish got. Right now the English engine handles 3rd-person-singular agreement, 25 irregular pasts, and quantifier-triggered plurals. That's enough for a basic sentence. It is not enough for the natural-feeling output Turkish already produces (purpose clauses, possessive agreement, dative for movement verbs). Closing that gap is the next push before any launch work.
